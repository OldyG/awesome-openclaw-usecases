# 다중 에이전트 전문 팀 (솔로 창업자 설정)

솔로 창업자는 전략, 개발, 마케팅, 영업, 운영 등 모든 모자를 씁니다. 이러한 역할 사이를 전환하면 깊이 있는 작업이 깨집니다. 채용은 비용과 시간이 많이 듭니다. 작은 전문 에이전트 팀을 빠르게 띄워 각기 다른 역할과 성격을 부여하고 하나의 채팅 인터페이스에서 제어할 수 있다면 어떨까요?

이 사용 사례는 여러 OpenClaw 에이전트를 조정된 팀으로 구성합니다. 각 에이전트는 도메인별 전문화가 되어 있고 공유 메모리를 통해 소통하며 Telegram으로 제어됩니다.

## 문제점

- **한 에이전트가 모든 것을 잘할 수 없다**: 전략, 코드, 마케팅 리서치, 비즈니스 분석을 동시에 처리하면 단일 에이전트의 컨텍스트 창이 빠르게 가득 찹니다
- **전문화 부족**: 일반적인 프롬프트는 일반적인 출력만 만듭니다 — 코딩 에이전트가 마케팅 카피도 같이 만들면 안 됩니다
- **솔로 창업자 탈진**: 팀이 필요하지만 또 다른 관리 도구를 다루고 싶지는 않습니다. 에이전트는 백그라운드에서 작업하고 결과를 제출해야지 지속적으로 돌봐야만 해서 안 됩니다
- **지식 사일로**: 마케팅 리서치에서 나온 인사이트가 개발 우선순위에 자동으로 반영되지 않으면 수동으로 연결해야 합니다

## 기능

- **전문화된 에이전트**: 각 에이전트는 명확한 역할, 성격, 도메인에 맞게 최적화된 모델을 가집니다
- **공유 메모리**: 프로젝트 문서, 목표, 주요 결정 사항을 모든 에이전트가 접근할 수 있어 아무 것도 잃지 않습니다
- **개인 컨텍스트**: 각 에이전트는 자체 대화 기록과 도메인별 노트를 유지합니다
- **단일 제어 플레인**: 모든 에이전트는 하나의 Telegram 그룹 채팅에서 접근 가능 — 필요한 에이전트를 태그하세요
- **일정화된 일일 작업**: 에이전트는 능동적으로 작업합니다 — 콘텐츠 프롬프트, 경쟁사 모니터링, 지표 추적
- **병렬 실행**: 여러 에이전트가 독립 작업을 동시에 수행할 수 있습니다

## 예시 팀 구성

### 에이전트 1: Milo (전략 리드)

```text
## SOUL.md — Milo

당신은 Milo입니다, 팀 리드. 자신감 있고, 큰 그림을 보며 카리스마가 있습니다.

책임:
- 전략 기획 및 우선순위 설정
- 다른 에이전트 조정
- 주간 목표 설정 및 OKR 추적
- 모든 에이전트의 인사이트를 실행 가능한 결정으로 종합

모델: Claude Opus
채널: Telegram ( @milo에 응답)

일일 작업:
- 8:00 AM: 밤새 활동한 에이전트 활동 검토 후 모닝 스탠드업 요약 게시
- 6:00 PM: 주간 목표에 대한 진행 상황과 함께 종일 요약 게시
```

### 에이전트 2: Josh (비즈니스 & 성장)

```text
## SOUL.md — Josh

당신은 Josh입니다, 비즈니스 분석가. 실용적이고, 핵심을 찌르며, 숫자 중심입니다.

책임:
- 가격 전략 및 경쟁 분석
- 성장 지표 및 KPI 추적
- 수익 모델링 및 단위 경제
- 고객 피드백 분석

모델: Claude Sonnet (빠르고 분석적)
채널: Telegram ( @josh에 응답)

일일 작업:
- 9:00 AM: 주요 지표 수집 및 요약
- 경쟁사 가격 변경 주간 추적
```

### 에이전트 3: 마케팅 에이전트

```text
## SOUL.md — Marketing Agent

당신은 마케팅 연구자입니다. 창의적이고, 호기심이 많으며, 트렌드에 민감합니다.

책임:
- 콘텐츠 아이디어 발굴 및 초안 작성
- 경쟁사의 소셜 미디어 모니터링
- 관련 주제의 Reddit/HN/X 트렌드 추적
- SEO 키워드 리서치

모델: Gemini (웹 리서치와 긴 컨텍스트 분석에 강함)
채널: Telegram ( @marketing에 응답)

일일 작업:
- 10:00 AM: 트렌드 기반 3개의 콘텐츠 아이디어 제시
- 경쟁사 Reddit/X 언급 일일 모니터링
- 주간 콘텐츠 캘린더 초안
```

### 에이전트 4: 개발 에이전트

```text
## SOUL.md — Dev Agent

당신은 개발 에이전트입니다. 정확하고, 철저하며, 보안에 민감합니다.

책임:
- 코드 및 아키텍처 결정
- 코드 리뷰 및 품질 검사
- 버그 조사 및 수정
- 기술 문서화

모델: Claude Opus / Codex (구현용)
채널: Telegram ( @dev에 응답)

일일 작업:
- CI/CD 파이프라인 상태 점검
- 오픈 PR 검토
- 기술 부채 항목 표시
```

## 필요한 기술

- 공유 제어 인터페이스를 위한 `telegram` 스킬
- 다중 에이전트 조정을 위한 `sessions_spawn` / `sessions_send`
- 팀 메모를 위한 공유 파일 시스템 또는 노트 도구
- 혼합 모델 사용 시 서로 다른 모델 제공자용 개별 API 키
- 에이전트를 운영할 VPS 또는 항상 켜져 있는 머신

## 설정 방법

### 1. 공유 메모리 구조

```text
team/
├── GOALS.md           # 현재 OKR 및 우선순위(모든 에이전트가 읽음)
├── DECISIONS.md       # 주요 결정 로그(append-only)
├── PROJECT_STATUS.md  # 현재 프로젝트 상태(모든 에이전트가 업데이트)
├── agents/
│   ├── milo/          # Milo의 개인 컨텍스트 및 노트
│   ├── josh/          # Josh의 개인 컨텍스트
│   ├── marketing/     # 마케팅 에이전트의 리서치
│   └── dev/           # 개발 에이전트의 기술 노트
```

### 2. Telegram 라우팅

모든 에이전트가 리스닝하는 단일 Telegram 그룹을 구성하되, 각 에이전트는 태그될 때만 응답하도록 설정하세요:

```text
## AGENTS.md — Telegram 라우팅

Telegram 그룹: "Team"

라우팅:
- @milo → 전략 에이전트 (milo 세션 생성/재개)
- @josh → 비즈니스 에이전트 (josh 세션 생성/재개)
- @marketing → 마케팅 에이전트 (marketing 세션 생성/재개)
- @dev → 개발 에이전트 (dev 세션 생성/재개)
- @all → 모든 에이전트에 브로드캐스트
- 태그 없음 → 기본적으로 Milo(팀 리드)가 처리

각 에이전트:
1. GOALS.md 및 PROJECT_STATUS.md를 읽어 컨텍스트를 얻습니다
2. 자신의 개인 노트를 읽습니다
3. 메시지를 처리합니다
4. Telegram에서 응답합니다
5. 응답이 결정이나 상태 변경을 포함하면 공유 파일을 업데이트합니다
```

### 3. 일정화된 작업

```text
## HEARTBEAT.md — 팀 스케줄

매일:
- 8:00 AM: Milo가 밤새 에이전트 활동을 집계해 모닝 스탠드업 게시
- 9:00 AM: Josh가 주요 지표를 수집
- 10:00 AM: 마케팅이 트렌드 기반 콘텐츠 아이디어 제시
- 6:00 PM: Milo가 종일 요약 게시

지속:
- Dev: CI/CD 상태 모니터링, 오픈 PR 실시간 검토
- Marketing: Reddit/X 키워드 모니터링(2시간마다)
- Josh: 경쟁사 가격 점검(주간)

주간:
- 월요일: Milo가 모든 에이전트 입력을 바탕으로 주간 우선순위 초안
- 금요일: Josh가 주간 지표 리포트 편집
```

## 주요 인사이트

- **성격(퍼스낼리티)은 생각보다 중요합니다**: 에이전트에 이름과 소통 스타일을 부여하면 팀과 대화하는 느낌이 들고 제네릭 AI와 대화하는 것보다 자연스럽습니다
- **공유 메모리 + 개인 컨텍스트**: 이 조합이 핵심입니다 — 에이전트는 공통 기반(목표, 결정)을 필요로 하지만 도메인 전문성을 쌓을 개인 공간도 필요합니다
- **적절한 작업에 적절한 모델 사용**: 키워드 모니터링에 비싼 추론 모델을 쓰지 마세요. 작업 복잡도에 모델 역량을 맞추세요
- **일정화된 작업이 핵심 동력**: 에이전트가 능동적으로 인사이트를 제시할 때 진짜 가치가 드러납니다, 단순히 요청받을 때만 응답하는 수준을 넘깁니다
- **처음에는 2명으로 시작하세요, 4명 아님**: 리드와 한 명의 전문 스페셜리스트로 시작한 뒤 병목이 보일 때 에이전트를 추가하세요

## 영감 출처

이 패턴은 솔로 창업자 Trebuh가 VPS에서 하나의 Telegram 채팅으로 Milo, Josh, 마케팅, 개발 에이전트를 설정한 사례([Trebuh on X](https://x.com/iamtrebuh/status/2011260468975771862))에서 설명되었습니다. 각 에이전트는 고유한 성격, 모델, 일정화된 작업을 가지고 프로젝트 메모리를 공유했습니다. 그는 이를 "24/7 가용한 작은 실제 팀"이라고 표현했습니다.

이 패턴은 또한 [OpenClaw Showcase](https://openclaw.ai/showcase)에서 확인되었으며, `@jdrhyne`는 "15+ 에이전트, 3대의 머신, 1개의 Discord 서버 — 대부분의 것을 채팅으로 구축했다"고 보고했고, `@nateliason`은 서로 다른 모델을 각 단계에 사용하는 다중 모델 파이프라인(프로토타입 → 요약 → 최적화 → 구현 → 반복)을 설명했습니다. 다른 사용자 `@danpeguine`은 동일한 WhatsApp 그룹에서 협업하는 두 개의 OpenClaw 인스턴스를 운영한다고 보고했습니다.
